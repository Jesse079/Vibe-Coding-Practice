# 决策记录

本文件记录已经确认、会影响后续开发的重要决定。若决定改变，追加新记录，不抹掉历史原因。

## 2026-06-22：首版使用 Vite 与原生 TypeScript

- 决定：不引入 React、Vue 等 UI 框架。
- 原因：首版只有单页画布和少量控件，原生 TypeScript 能减少依赖、状态层级和维护成本。

## 2026-06-22：使用 MediaPipe Hand Landmarker

- 决定：通过 `@mediapipe/tasks-vision` 获取单手 21 个关键点。
- 原因：适合浏览器本地实时处理，能直接支持手势几何判断，不需要自建模型服务。

## 2026-06-22：单手、本地、桌面 Chrome 优先

- 决定：首版只追踪一只手，只保证桌面 Chrome 的本地使用。
- 原因：先缩小变量，验证核心交互，再考虑跨浏览器、移动端和多手。

## 2026-06-22：三种互斥模式

- 决定：应用状态固定为 `idle`、`configuring`、`drawing`。
- 原因：通过显式状态机避免配置选择、绘制和清除之间互相误触。

## 2026-06-22：捏合采用相对距离和双阈值

- 决定：拇指与食指距离按手掌尺度归一化，并使用不同的进入和松开阈值。
- 原因：适应手与摄像头距离变化，减少临界值附近反复抖动。

## 2026-06-22：开发日志按会话收尾更新

- 决定：使用 `dev-logs/YYYY-MM-DD.md`，每次真实开发任务结束时创建或追加。
- 原因：系统定时任务无法准确知道当天完成了什么，按会话记录更可靠。

## 2026-06-22：先测量，再决定是否使用 Web Worker

- 决定：Hand Landmarker 初期在主线程运行；出现可测量的持续卡顿后再迁移 Worker。
- 原因：Worker 会增加视频帧传输、资源加载和调试复杂度，不应在没有性能证据时提前引入。

## 2026-06-22：选定“沉浸舞台”视觉方向

- 决定：采用方案 1，使用全屏画布、顶部轻量状态、底部横向工具坞和上浮配置面板。
- 视觉基准：`docs/assets/immersive-stage-reference.png`。
- 原因：它把最大面积留给创作，同时让当前画笔、操作提示和清除入口保持可见，结构直观且实现风险低于环形空间菜单。
- 约束：参考图只定义视觉层级和氛围；正式手势含义继续以需求和交互规范为准。

## 2026-06-23：MediaPipe 资源本地化

- 决定：锁定 `@mediapipe/tasks-vision` 0.10.35，并将官方 Hand Landmarker 模型和 WASM 放进 `public/`。
- 原因：应用运行不依赖第三方 CDN，版本和资源来源可追踪。

## 2026-06-23：GPU 优先、CPU 回退

- 决定：Hand Landmarker 优先使用 GPU delegate，初始化失败时自动回退 CPU。
- 原因：优先降低实时绘制延迟，同时兼容不支持 GPU 路径的设备。

## 2026-06-23：使用 Vitest 验证核心逻辑

- 决定：新增 Vitest，仅测试状态机、手势滞回、计时、映射和笔画数据等稳定逻辑。
- 原因：这些逻辑如果只依赖摄像头手工测试，容易出现回归且难以复现。
