# AGENTS.md

## 项目目标

本项目是一个本地运行的“隔空手写画板”网页。它通过电脑摄像头识别单手关键点，让用户用手势配置画笔、绘制轨迹和清空画布。

默认目标是：用最稳、最简单、最少改动的方式完成当前里程碑，不提前实现后续功能。

## 开始工作前

按以下顺序阅读：

1. `docs/01-requirements.md`：确认功能范围和不做事项。
2. `docs/02-technical-design.md`：确认技术方案和模块边界。
3. `docs/03-interaction-design.md`：确认模式、手势和防误触规则。
4. `docs/04-development-plan.md`：确认当前里程碑及验收条件。
5. `docs/05-coding-standards.md`：确认代码和目录规范。
6. `docs/06-testing-and-acceptance.md`：确认本阶段需要执行的检查。
7. `docs/07-security-and-privacy.md`：确认摄像头和敏感数据规则。
8. `docs/08-decisions.md`：了解已经确定的重要技术决策。
9. `dev-logs/` 中日期最新的日志：了解已完成事项、问题和下一步。

`docs/README.md` 是文档索引，标准发生变化时应同步维护。

## 工作规则

1. 一次只处理 `docs/04-development-plan.md` 中当前里程碑。
2. 当前里程碑验收通过后，才能开始下一里程碑。
3. 修改前先查看相关文件和 Git 状态，不覆盖用户已有改动。
4. 保持原有代码风格、命名和目录结构，不做无关重构。
5. 不随意增加依赖；确需新增时，先说明用途和更简单的替代方案。
6. 修复问题时找到原因，不通过关闭安全检查或吞掉错误来掩盖问题。
7. 修改公共类型、状态事件或模块接口时，同步检查所有调用方。
8. 修改需求、技术或交互约定时，同步更新对应 `docs/` 文件。
9. 重要技术取舍写入 `docs/08-decisions.md`，包括日期、决定和原因。
10. 完成任务后执行与改动相匹配的检查，并把结果写入当日日志。

## 开发日志

- 日志目录：`dev-logs/`
- 文件名：使用电脑本地日期，格式为 `YYYY-MM-DD.md`
- 同一天的后续任务追加新记录，不覆盖原记录。
- 每次开发任务收尾时必须记录：
  - 本次目标
  - 完成事项
  - 修改文件
  - 验证结果
  - 发现的问题
  - 下一步待办
- 只有实际完成或验证过的内容才能写入“完成事项”。
- 未完成事项保留在“下一步待办”，后续完成后再明确标记。

日志不是系统定时生成，而是在每次真实开发会话结束时自动创建或追加，避免产生不准确的空日报。

## 安全要求

1. 摄像头画面和手部关键点只在用户浏览器本地处理。
2. 不上传、存储或记录摄像头画面。
3. 不在代码、日志或文档中写入 API Key、Token、密码、Cookie 或私钥。
4. 不提交 `.env`、证书、密钥或包含个人画面的调试材料。
5. 不绕过浏览器摄像头权限，不伪装权限状态。
6. 不使用危险的递归删除命令；删除重要文件前必须得到用户明确授权。

## 完成汇报

每次任务完成后用中文简要说明：

- 改了哪些文件
- 解决了什么问题
- 做了哪些验证
- 仍需注意什么
